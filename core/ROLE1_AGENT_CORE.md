# 🧑‍💻 角色 1：Agent 核心与提示词工程师 (Agent Core & Prompt Engineer)

## 🎯 核心目标
你的任务是打造 Agent 的“大脑”。你需要确保大语言模型（LLM）能够理解 CTF 的目标，合理地拆解任务，正确地调用工具，并且在遇到挫折时能够自我反思和修正，而不是陷入死循环。

## 📁 负责目录
`my-ctf-agent/core/`

## 🏗️ 架构设计与实现指导

### 1. 核心循环设计 (agent.py)
* **建议**：基于 `langchain` 或 `langgraph` 构建主循环。MVP 阶段可以直接使用 `create_agent` (ReAct 范式)。
* **关键点**：确保模型能够无缝接收来自角色 2 (MCP) 和角色 3 (本地工具) 的所有工具。

### 2. 提示词工程 (prompts.py)
CTF 场景非常特殊，模型容易“想太多”或“乱用工具”。你的 System Prompt 必须包含以下结构：
* **角色设定**：专业的 CTF 安全专家。
* **工作流约束 (SOP)**：强制模型遵循 `信息收集 -> 漏洞分析 -> 利用 -> 提交 Flag` 的流程。
* **工具使用守则**：明确告诉模型什么时候该用什么工具。例如：“当面对网页时，先用 `http_get` 抓取源码，而不是直接上 `sqlmap`”。
* **少样本示例 (Few-Shot)**：给模型提供 1-2 个成功的解题对话历史作为参考。

### 3. 记忆与状态管理 (memory.py)
* **痛点**：在多轮对话后，历史记录（Message History）会变得非常巨大，不仅消耗 Token，还会让模型“走神”。
* **设计指导**：
  * 实现一个自定义的 `AgentState`，在状态中单独提取并维护“当前发现的线索 (Findings)”和“已尝试的 Payload”。
  * 定期总结历史对话，将长对话折叠为摘要，释放 Context Window。

### 4. 容错与防死循环中间件 (middlewares.py)
* **痛点**：Agent 经常会连续几十次调用同一个工具（比如不断用同样的密码尝试 SSH 登录失败）。
* **设计指导**：
  * 在工具调用的中间件层（参考原模板的 `wrap_tool_call`）记录工具调用的频率和参数。
  * 如果发现 `工具A` 连续 3 次使用相同参数返回相同的报错，强行向大模型注入提示：“警告：你正在重复相同的错误操作，请立即停止并尝试其他思路，或查阅专家知识库。”

## 🤝 协作指南
* 向 **角色 3** 提出工具需求（例如：“现在的命令执行工具返回结果太长了，经常爆 Token，请帮我截断一下”）。
* 与 **角色 5** 合作，将专家的解题思路融入到你的 Prompt 中。
