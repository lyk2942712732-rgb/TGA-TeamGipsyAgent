## 暂缓事项

### 1. 上下文摘要压缩

- [ ] 评估是否需要 LLM 摘要型上下文压缩。
- [ ] 明确摘要不能替代 Blackboard，不能把未验证推测写成已确认事实。
- [ ] 设计摘要触发条件，例如消息数、token 估算、step 数或 checkpoint 大小。
- [ ] 将摘要结果作为独立记忆字段保存，而不是混入 `confirmed_facts`。
- [ ] 与当前 `recent_messages` 窗口裁剪和 Blackboard 结构化状态协调。

说明：当前已有消息窗口裁剪和 Blackboard 状态压缩，能满足 MVP 的基础上下文控制；真正的“定期总结历史对话”可等长任务联调后再做。

### 2. Prompt 工具注入策略

- [ ] 明确 Core 只负责接收和绑定工具，不在 Agent Core 中硬编码具体工具能力。
- [ ] 等角色 2 MCP 工具和角色 3 安全工具稳定后，设计“可用工具摘要”注入格式。
- [ ] 从工具 `name`、`description`、参数 schema 中提取简短工具说明。
- [ ] 在 prompt 中强调“只使用当前注入工具”，而不是预设所有工具一定存在。
- [ ] 避免把某个阶段固定绑定到不存在的工具，例如强制调用未注入的 `http_get`。

说明：工具使用守则应由实际注入工具动态生成；Core 目前接口已经留好，暂不需要提前写死工具清单。

### 3. 功能测试

- [ ] Fake Model + Fake Tool：工具返回 Flag 后，LLM 复述并进入等待用户提交状态。
- [ ] 用户在同一 `thread_id` 回复“提交成功”后，状态变为 `completed`。
- [ ] 用户回复“提交失败”后，错误 Flag 被标记失败并继续解题。
- [ ] 连续三次相同工具、相同参数、相同错误后，后续调用被 Guard 拦截。
- [ ] 即使外部传入 `submit_flag`，模型绑定工具中也不应包含它。
- [ ] 不同 `thread_id` 的 Blackboard 互相隔离。
- [ ] 针对上下文摘要压缩和动态工具注入补充专项测试。

说明：自动化验收需要角色 4 的测试框架、依赖环境和 Fake/Mock 设施配合；当前先记录用例方向。

### 4. Flag 候选显示注入

- [ ] 评估是否在 Blackboard prompt 中显示待复述 Flag 的原始值。
- [ ] 如果显示，必须确保值来自工具原始输出，且不经过大小写改写、脱敏或截断。
- [ ] 需要避免把已被用户判定失败的 Flag 再次作为待提交候选注入。
- [ ] 设计多候选 Flag 的展示顺序和数量上限。
- [ ] 增加测试覆盖：历史消息被裁剪后，LLM 仍能看到待复述的候选 Flag。

说明：当前实现会提示“工具中已出现 Flag”，但不直接展示候选值；短流程通常依赖最近 ToolMessage 即可，长流程裁剪后再补这个增强。

1. 等角色 2/3 工具接口稳定后，做动态工具说明注入。
2. 等角色 4 测试环境可用后，补齐 Core 最小验收测试。
3. 在长任务上下文压力真实出现后，再实现摘要型上下文压缩。
4. 如果发现历史裁剪导致 LLM 无法复述 Flag，再补 Flag 候选值显示注入。
