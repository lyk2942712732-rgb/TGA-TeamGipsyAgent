# 🕵️‍♂️ 角色 5：CTF 赛道专家 (Vulnerability Specialist)

## 🎯 核心目标
你的任务是充当 Agent 的“师傅”。大语言模型虽然读过很多书，但实战经验往往不足（容易产生幻觉，或者瞎猜）。你需要把人类打 CTF 的宝贵经验（SOP、字典、Payload、定制脚本）沉淀下来，教给 Agent。

## 📁 负责目录
`my-ctf-agent/expert_knowledge/`

## 🏗️ 架构设计与实现指导

### 1. 编写解题 SOP (Standard Operating Procedures)
* **痛点**：大模型碰到一个 Web 题，可能上来就盲目扫目录，或者直接用 sqlmap 跑全站，效率极低且容易报错。
* **设计指导**：
  * 用 Markdown 编写特定赛道（如 Web SQLi, Web SSRF, Crypto）的“人类解题攻略”。
  * 攻略必须是**结构化**、**可执行**的。例如《SQL注入处理指南》：
    1. 观察页面特征（是否有传参如 `?id=1`，是否有登录框）。
    2. 手动探测（指导模型使用单引号 `'`、双引号 `"`、反斜杠 `\` 观察是否报错）。
    3. 工具利用（明确告诉模型，什么情况下才去调用自动化工具，以及应该传什么参数）。
  * 这些 Markdown 文件后续会被 **角色 1** 通过 RAG 或者动态 Prompt 注入给 Agent。

### 2. 提供“核武器” Payload 和 专属字典
* **设计指导**：
  * 模型自己编的 Payload 往往有语法错误或不够巧妙。你需要在文档中列举出常见的高效 Payload（如万能密码、常见的过滤绕过方式）。
  * 如果需要，整理一份极简的高质量弱口令字典/目录扫描字典放在当前目录下，供 Agent 调用。

### 3. 定制化辅助脚本 (custom_scripts/)
* **设计指导**：
  * 某些漏洞的利用过程过于复杂，如果让大模型自己一步步操作（比如涉及多次加密解密、特定的 Cookie 伪造序列），它肯定会出错。
  * 作为专家，你应当把这些复杂流程写成一个 Python 脚本（比如 `exploit_oracle_padding.py`）。这个脚本对外暴露极简的参数（只需输入 URL）。
  * 写好后，交给 **角色 3**，让他包装成 Agent 可以直接调用的工具。

## 🤝 协作指南
* 与 **角色 1** 配合，不断审查 Agent 的执行日志，分析“为什么它没有按我的思路去解题”，并修改你的 SOP 文档。
* 与 **角色 3** 配合，把你的自动化脚本转换为规范的 LangChain 工具。
