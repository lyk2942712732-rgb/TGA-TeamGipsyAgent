**角色说明：**

###  角色 1：Agent 核心与提示词工程师 (Agent Core & Prompt Engineer)

**职责目标**：赋予 Agent “大脑”，确保它能正确思考、规划任务，且不会陷入死循环。 **本周 MVP 任务**：

1. **接管主循环**：基于 `agent-template/main.py` 中的 `create_ctf_agent`，负责 Agent 的整体实例化和运行逻辑。
2. **Prompt 工程**：深度优化 `SYSTEM_PROMPT`。由于 CTF 需要分步探测（信息收集 -> 漏洞分析 -> 漏洞利用 -> 提权/拿 Flag），需要设计结构化的 Prompt 和 Few-Shot（少样本）示例。
3. **状态与记忆管理**：扩展 `CTFAgentState`，确保 Agent 能够记住之前尝试过的 Payload，避免重复无效攻击。
4. **防死循环机制**：在中间件（Middleware）中加入逻辑，当发现 LLM 连续多次调用同一工具且报错时，强制干预或中断。

### 🔌 角色 2：平台对接与 MCP 工程师 (Platform Integration & MCP Dev)

**职责目标**：赋予 Agent “沟通能力”，确保它能与 CTF 平台完美交互。 **本周 MVP 任务**：

1. **MCP 客户端调试**：使用 `langchain-mcp-adapters`，确保与平台的 MCP Server (SSE 传输) 建立稳定连接。
2. **平台工具打通**：跑通平台提供的标准流程工具，包括：获取题目列表 (`get_challenges`)、启动靶机 (`start_challenge`)、提交 Flag (`submit_flag`)。
3. **认证与鉴权**：处理环境变量中的 `CTF_TOKEN` 等鉴权信息的传递。
4. **异常处理**：处理网络抖动、靶机启动超时等平台级异常，通过拦截器向 Agent 屏蔽底层网络错误，转换为友好的自然语言提示。

### 🧰 角色 3：安全工具链工程师 (Security Toolchain Engineer)

**职责目标**：赋予 Agent “双手”，开发供 Agent 调用的攻击和探测武器库。 **本周 MVP 任务**：

1. **扩展基础工具**：接手 `agent-template/tools/` 目录，将现有的 `http_tools.py` 升级为支持复杂请求（自定义 Header、Cookie、POST Body 等）的工具。
2. **代码执行环境**：为 Agent 开发一个安全的本地终端/Python 执行工具（例如基于 `subprocess` 的 bash 执行器，或 Python REPL），以便 Agent 能够随时编写和运行简单的攻击脚本（如用 `requests` 发包）。
3. **信息截断与清洗**：Agent 极其容易因为工具返回的输出过长（例如一个巨大的 HTML 页面或 Nmap 结果）而导致 Token 超出限制 (Context Window Overflow)。你需要编写工具包装器，自动提取关键信息或截断过长输出。

### 🏗️ 角色 4：测试与基础设施工程师 (Infra & Testing Engineer)

**职责目标**：确保 Agent 跑得稳、能打包、好调试。 **本周 MVP 任务**：

1. **容器化打包**：维护 Agent 的 `Dockerfile` 和依赖（`uv.lock`），确保 Agent 能够被顺利打包推送到平台的 Docker Registry 中被调度。
2. **可视化与日志**：实现 `main.py` 中的 `log_before_model` 和 `log_after_model` 中间件，将 Agent 的“思考过程”和“工具调用”打印成易读的格式，或者接入 LangSmith 以便团队复盘 Agent 是怎么“死”的。
3. **本地 Mock 测试环境**：为了不依赖主平台，在本地使用 Docker 启动一个简单的漏洞靶机（如项目中的 `challenges/web-sql-injection`），供团队其他成员在本地无缝测试 Agent，提高开发效率。

### 🕵️‍♂️ 角色 5：（可选）CTF 赛道专家 (Vulnerability Specialist)

*注：如果团队只有 4 人，该角色由 角色 1 或 角色 3 兼任。* **职责目标**：为 MVP 挑选一个具体的突破口，帮助 Agent 拿下“第一滴血”。 **本周 MVP 任务**：

1. **聚焦单一赛道（如 Web SQLi）**：对于 MVP 阶段，不要指望 Agent 什么题都能做。挑选平台中现有的某一道靶机（如 SQL 注入），整理出人类视角的“标准解题路径（Happy Path）”。
2. **提供专家级 Payload**：把人类的解题思路转化为 Agent 能理解的工具和提示词（例如，告诉 Agent 测试 SQLi 时应该优先尝试哪些闭合符）。
3. **验收测试**：扮演测试员，不断用这道题目“考核” Agent，记录其失败的原因并反馈给核心开发（角色 1 和 3）。
